AI 对话上下文管理设计

核心原则：113 原则
元素	数量	说明
最近对话	1 轮	最近 1 对（用户提问 + AI 回复），保持对当前话题的即时响应
最近摘要	1 个	最近的已生成摘要，提供对话的连续性和关键信息
摘要触发周期	3 轮	每 3 轮对话触发一次摘要生成（不包括当前最新轮）
上下文结构
┌─────────────────────────────────────────────┐
│  上下文 = [最近摘要] + [最近1轮对话]          │
│                                             │
│  示例（假设当前是第7轮对话）：                  │
│                                             │
│  摘要（第3轮后生成）✔                          │
│  第7轮（用户提问+AI回复）✔                     │
│  第6轮（用户提问+AI回复）✘（被取代）            │
│  第1-5轮（已压缩为摘要）✘                     │
└─────────────────────────────────────────────┘
触发机制
触发条件（在用户提交问题后）：

用户提交新问题
↓
检查当前对话轮数 N（从 1 开始计数）
↓
如果 N > 3 且 (N - 1) % 3 === 0
  → 触发摘要生成（将最近 3 轮对话压缩为摘要）
↓
否则
  → 不触发
示例流程：

─────┼─────────────┼──────────
  1  │  否         │ []
  2  │  否         │ []
  3  │  ✔（用户提交第4个问题后触发） │ [摘要1]

  9  │  ✔          │ [摘要1, 摘要2, 摘要3]
  ...
关键规则：

触发时机：用户提交问题后、AI 回复之前
触发轮次：第 N 轮对话完成后（即用户提交第 N+1 个问题时），如果 N > 3 且 (N - 1) % 3 === 0
摘要对象：最近 3 轮对话（不含当前最新轮）
摘要存储：仅保留最近 1 个摘要，旧摘要可被覆盖或合并
摘要生成
Prompt 设计
const SUMMARY_PROMPT = `将以下对话压缩为后续对话上下文。

要求：
1. 保留主题、结论、未解决问题和关键约束
2. 删除举例、解释过程和重复内容
3. 使用最简洁的表达
4. 摘要控制在300字以内
5. 提取3-5个关键词
6. 不要添加原文没有的信息

对话：
{conversation}

输出格式：
摘要：<摘要内容>
关键词：<关键词1>、<关键词2>、<关键词3>`;
调用流程
用户提交新问题
↓
检查是否触发摘要生成
↓
是 → 收集最近3轮对话（去掉当前最新轮）
     ↓
     调用 DeepSeek API（stream=false）
     ↓
     解析返回的摘要和关键词
     ↓
     更新上下文缓存 + 持久化到数据库
     ↓
     继续处理用户问题
↓
否 → 直接继续处理用户问题
异步处理：

摘要生成采用同步阻塞方式（在用户提交后、AI 回复前完成）
如果 API 调用失败，使用简单的文字截断作为 fallback
摘要生成时间建议 < 2 秒，否则用户感知到延迟
数据模型
// Zustand Store
interface ChatState {
  messages: Message[];
  summaries: ChatSummary[]; // 最多保留 1 条
  context: {
    summary: string | null;      // 最近摘要
    keywords: string[] | null;    // 最近关键词
    lastSummarizedMessageId: string | null; // 上次摘要最后一条消息的 ID（通常是 AI 回复）
  };
}

// 数据库存储（必须持久化）
interface ChatSummary {
  id: string;                    // 主键
  chat_id: string;               // 所属对话 ID
  content: string;               // 摘要内容（≤300字）
  keywords: string[];            // 3-5 个关键词
  last_message_id: string;       // 摘要中包含的最后一条消息 ID（通常是 AI 回复 ID）
                                 // 用于判断是否需要更新摘要
  created_at: string;            // 创建时间
}
last_message_id 的使用规则：

当需要判断是否触发新摘要时：
1. 查询 ChatSummary 表，获取当前对话的最新记录
2. 如果 last_message_id 对应轮次 < 当前轮次 - 3
   → 触发新摘要
3. 新摘要生成后，将 last_message_id 更新为最新的 AI 回复 ID
用户无感设计
界面上不显示任何摘要相关元素
摘要仅在内存和数据库中作为上下文使用
用户无法查看、编辑或删除摘要
性能影响控制在 100ms 以内（缓存 + 本地数据库操作）
API 调用失败时自动降级（使用基础上下文）
边界情况处理
情况	处理方式
摘要生成失败	使用上次成功摘要，不阻塞对话
深度求索 API 超时	设置 5 秒超时，超时后跳过本次摘要
摘要长度超限	截断至 300 字，保留首尾
关键词提取失败	使用默认关键词（"对话"、"问题"、"讨论"）
对话轮次不足 3 轮	使用完整对话作为上下文
首次触发延迟	首次触发在用户提交第 4 个问题后，不影响前 3 轮
last_message_id 为空	从头开始摘要（首次生成）
